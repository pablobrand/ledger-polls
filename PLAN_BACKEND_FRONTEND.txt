Project plan: Backend API, Frontend, and Integration for Ledger-Polls

Overview

This document is a working plan of action to finish the backend API for the survey system (wallet-auth + encrypted PII), implement the frontend flows (Phantom wallet, DEK/KEK wrapping, persona creation, survey-taking), and wire them together with CI and migrations. It assumes the current Prisma schema (backend/prisma/schema.prisma) is the canonical DB model.

Goals
- Backend: secure REST API for wallet-auth, persona storage (encrypted), consent and matching, surveys management, assignments, answers (encrypted), payouts, and audit logging.
- Frontend: React + Vite app that provides Phantom wallet login, DEK generation & wrapping, persona creation UI, survey discovery and answer submission with client-side encryption, and consent flows.
- Integration: Docker-based local dev, reproducible CI pipeline (generate Prisma client, run migrations, build frontend), and E2E smoke tests.

Constraints & Security
- The server must never hold unwrapped DEKs or raw PII except ephemeral signed submission flows from the user (preferably avoid sending plaintext to server).
- Matching should use non-PII hashed traits (UserTrait) stored in DB.
- All encrypted payloads must be envelopes (alg, ct, iv, tag, v, aad optional).
- Use server-generated nonces/challenges on signature flows to prevent replay.
- Recovery: if user loses wallet key, data is irrecoverable unless they exported a recovery key or opted into a separate recovery wrap.

High-level components
- Backend (Node.js, Express, Prisma): auth controllers, user service, keywrap service, persona endpoints, survey admin endpoints, assignment engine, answer submission endpoint, payouts, audit log.
- Frontend (React + Vite): wallet adapter integration, crypto utilities (SubtleCrypto helpers), persona UI, survey UI, answer encryption & submission, consent dialogs.
- CI: GitHub Actions example in README; add steps to validate Prisma and build artifacts.

Priority roadmap (milestones)
1. Core backend scaffolding and auth (wallet login + user creation) — Highest priority
2. Client-side crypto helpers & wallet sign/derive KEK sample — parallel with (1)
3. Persona creation flow + store encrypted persona and wrapped DEK — both sides
4. Survey CRUD & eligibility rule model support (admin/Org) + matching (read-only engine using hashed traits)
5. Assignment engine (generate SurveyAssignment rows for eligible users)
6. Answer submission (encrypted) + trait promotion to UserTrait
7. Payout handling + audit logging + basic admin UI endpoints
8. Frontend polish, UX, and end-to-end verification
9. CI pipeline and migration hardening

Detailed tasks (backend)
1. Auth & User
   - Endpoint: POST /api/auth/login — accepts walletAddress, returns server nonce and ephemeral session token
   - Endpoint: POST /api/auth/verify — accepts walletAddress + signature over nonce, creates or returns user record
   - Create WalletAuth service to store lastSignedAt, lastNonce
   - Create server-side session token issuing (JWT or signed cookie) for API calls
   - Acceptance: user row created with walletAddress and generatedUserId; session returned

2. KeyWrap & Persona
   - Endpoint: POST /api/user/persona — authenticated user uploads encrypted persona envelope and wrappedKey metadata
   - Backend stores envelope in Persona or EncryptedBlob and stores wrappedKey in UserKeyWrap
   - Endpoint: GET /api/user/persona — returns envelope metadata and whether decryption is possible (consent)
   - Acceptance: server does not receive plaintext DEK; wrappedKey stored as Json

3. Consent
   - POST /api/user/consent — record scope and consent
   - GET /api/user/consent
   - Use Consent in authorization checks where required

4. Surveys & Eligibility
   - Admin endpoints for Org to create surveys and eligibility rules: CRUD routes (protected)
   - Implement normalized Eligibility rows for matching against hashed traits (UserTrait)
   - Add an engine to evaluate eligibility by joining hashed traits and survey eligibility rows
   - Acceptance: admin can create a survey and see matched user count (approx)

5. Assignments
   - Worker or API route to assign eligible users to surveys (respecting quotas and maxPerUser)
   - Insert SurveyAssignment rows; send notification (placeholder)

6. Answers
   - Endpoint: POST /api/surveys/:surveyId/assignments/:assignmentId/answers
   - Body: encrypted envelope (value), questionId; server stores Answer row with encrypted JSON
   - If client indicates promoted: compute trait(s) client-side or accept promotion metadata from client in hashed form and write to UserTrait
   - Acceptance: answers stored encrypted; unique constraint per assignmentId+questionId enforced

7. Payouts & Audit
   - Payout creation/claim endpoints, admin payouts list
   - Audit logging helper — write AuditLog rows for login, assign, submit, decrypt-request

8. Tests & Migrations
   - Add unit tests for services (non-crypto parts): user creation, survey creation, assignment logic
   - Create initial migration script using prisma migrate dev --create-only then review SQL if DB contains data

Detailed tasks (frontend)
1. Wallet + auth
   - Integrate @solana/wallet-adapter (Phantom) or the project's chosen adapter
   - Login flow: request server nonce, sign with wallet, send signature to verify
   - Persist session token securely (httpOnly cookie or local storage with CSRF protections)

2. Crypto helpers
   - Implement helper module (web crypto wrapper) to:
     - Generate DEK (AES-256-GCM)
     - Encrypt/decrypt envelopes with DEK
     - Derive KEK from wallet signature + challenge using HKDF-SHA256
     - Wrap/unwrap DEK with KEK
   - Provide example utilities and small unit tests (Jest)

3. Persona flow
   - UI to collect persona fields (form)
   - On submit: generate DEK (if not present), encrypt persona fields, request server nonce, sign, derive KEK, wrap DEK, send envelope + wrappedKey
   - Show consent checkbox for MATCHING and DECRYPT scopes

4. Survey UX
   - Survey listing, question rendering (supports types in QuestionType), option selection
   - On submit: encrypt answers with DEK and POST to answers endpoint
   - Support branching logic: client-side skip logic as specified in Question.logic

5. Decrypt & consent flows
   - When org requests decryption: show request to user (modal) — user signs new nonce, unwraps DEK, and approves whatever minimal plaintext is requested
   - Implement client-side aggregation to avoid sending bulk plaintext to server

6. Offline handling & multi-device
   - If user logs in on a new device, provide rewrap flow: client re-derives KEK after signing, unwraps DEK (if previous device provided DEK export) or request user to re-enter persona

Integration & CI
- Add to CI (GitHub Actions):
  - Backend: npm ci, npx prisma validate, npx prisma generate
  - Frontend: npm ci, npm run build
  - Optionally run unit tests and lint
- Docker: ensure Dockerfile runs npx prisma generate during build and that migrations are applied with migrate deploy on container startup or CI
- Migration strategy in CI: run `npx prisma migrate deploy` against prod DB; in dev use migrate dev

Migration practical notes
- If migrating an existing DB with rows, prefer creating migration SQL manually via `--create-only` and add defaults/backfill steps where required.
- For new development DBs, `npx prisma migrate dev` is fine.

Acceptance criteria (per milestone)
- Auth: user can sign in with Phantom and receive a session token; server stores walletAddress
- Persona: user can create persona; server stores encrypted envelope + wrappedKey; no plaintext leaks to server
- Survey pipeline: admin creates survey; system assigns users; user sees assignment and submits encrypted answers
- Matching: server can compute eligible users using UserTrait hashed values
- CI: `npx prisma validate` and `npx prisma generate` run successfully in CI
